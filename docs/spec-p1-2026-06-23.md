# 鸿蒙空间化商品 3D 展示 App · P1「空间核心闭环」设计

- **日期**：2026-06-23
- **状态**：设计已与用户对齐，待评审
- **目标系统**：HarmonyOS 7（API 26）
- **能力依据**：沉浸光感组件、Spatial Recon Kit（3DGS 端侧重建）

---

## 1. 背景与愿景

基于 HarmonyOS 7（API 26）的「空间化：从平面到立体」能力，构建一个面向**小件精品**（饰品、潮玩、手办、珠宝、玻璃摆件）的 3D 商品展示平台。商家用手机环绕商品扫描，端侧 3DGS 实时重建出三维模型；买家在沉浸光感材质渲染下旋转、缩放、端详商品细节。

**差异化论点**：支付/目录/订单任何框架都能做，但「端侧 3DGS 重建 + 沉浸光感材质」的组合只有 HarmonyOS 7 有。这是本 App 唯一不可替代的卖点。

### 产品定位（已与用户对齐）

| 维度 | 选择 |
|---|---|
| 方向 | 商品 3D 展示 |
| 模型来源 | **商家现场扫描，重建为核心** |
| 商品品类 | 小件精品 |
| 交易范围 | 完整电商闭环（**完整平台愿景**） |
| 架构 | **单 App 双角色**（买家⇄商家模式切换） |

### 平台全景与分期

完整电商闭环是 3–4 个子项目的体量，按以下顺序分期建设，**每期一份独立 spec → plan → 实现**：

| 阶段 | 名称 | 内容 |
|---|---|---|
| **P1（本设计）** | 空间核心闭环 | 扫描重建 → 存储 → 沉浸展示 → 光感互动。**不碰交易**。 |
| P2 | 商品化与目录 | 商家上架流程、商品化包装、分类搜索。模型变成"商品"。 |
| P3 | 交易闭环 | 购物车、华为支付、数字盾（TEE 级）、订单售后。 |
| P4 · v2 | 分发增长 | 消费侧抽成元服务，服务卡片/闪控窗/平行视界 easygo 分发。 |

## 3. 架构与数据流

### 技术分层

- **NDK 层（C/C++，紫色）**：AR Engine 会话、Spatial Recon 重建、3DGS 模型加载渲染
- **ArkTS 层（绿色）**：UI、沉浸光感材质、捕获引导、导航、ModelStore
- **桥接**：ArkTS ↔ NDK 通过 **napi**

### 端到端数据流

```
Camera 取帧(1080×1440 RGB) → AR Engine(位姿/内参)
  → Spatial Recon PushARFrame → 3DGS 重建
  → 落盘 PLY(+可选 MP4 运镜视频)
  → ModelStore 索引(名称/品类/时间/缩略图)
  → 3DGS loader 加载 → XComponent 渲染
  → 沉浸光感 chrome 覆盖(光随指动/光线勾勒/非线性形变)
```

**硬约束**：
- 输入图像必须 1080×1440 RGB，其余尺寸结果未定义。
- 同一时刻只允许一个 Spatial Recon session 进行重建，否则未定义行为。
- 重建消耗大，必须订阅温升事件并按需暂停。

## 4. 模块详解

五个模块，单一职责、显式接口、可独立测试。

### ① CaptureKit（扫描捕获）· NDK

管 Camera + AR Engine 会话；推 1080×1440 RGB 帧给 Spatial Recon；订阅温升事件。

- **接口**：`start() / stop() / pushFrame()`
- **帧来源**：使用 AR Engine 数据结构（`HMS_AREngine_ARSession` + `ARFrame`），调 `HMS_AREngine_ARSession_Update` 后用 `HMS_SpatialRecon_PushARFrame` 推入。理由：AR Engine 免费给出相机内参+位姿，省去手拼 `focalX/focalY/principalX/principalY`。
- **预览尺寸**：`HMS_AREngine_ARConfig_SetPreviewSize(1080, 1440)`。
- **RunningMode**：前台扫描时 `SPATIAL_RECON_RUNNING_FOREGROUND_MODE`，应用切后台立刻切 `BACKGROUND`。设错会性能/功耗劣化。

### ② ReconEngine（重建引擎）· NDK

封装 Spatial Recon 生命周期、进度、产物保存。**单 session 守卫**——并发 `start()` 直接拒绝。

- **接口**：`start() / pause() / resume() / progress() / save()`
- **生命周期**：`Create → PushFrame（可重复）→ StartSession → [Pause/Resume] → Save`。`StartSession` 传入非空 `ModelWriteInfo` 时重建完成自动保存。
- **进度**：`HMS_SpatialRecon_GetProgress` 以 1–2Hz 轮询，驱动进度环。
- **关键帧**：系统自动选取（`PushFrame` 内部），开发者不参与。
- **产物**：
  - **PLY 点云**（主）——给 Showcase 加载渲染
  - **MP4 运镜视频**（可选）——本地保存的预览片段，P1 不做分享

### ③ ModelStore（模型管理）· ArkTS

本地落盘 + 索引；上传接口 stub。

- **接口**：`save() / list() / load() / delete()`
- **索引**：RelationalStore，字段 = {id, 名称, 品类, 创建时间, 缩略图路径, PLY 路径, 材质预设}
- **文件**：PLY/MP4 写入 app sandbox
- **上传**：P1 stub（签名固定返回本地路径占位），P2 接真实对象存储 + CDN

### ④ Showcase（沉浸展示）· ArkTS + NDK

3DGS 渲染 + 手势 + 光感材质。**核心差异化所在。**

- **接口**：`loadModel() / setMaterial() / resetView()`
- **渲染器**：Spatial Recon Kit 自带「3DGS 模型加载器」，经 `XComponent` 取 native surface。
- **手势**：单指拖拽→旋转 ｜ 双指捏合→缩放 ｜ 双指拖拽→平移 ｜ 双击→复位
- **光感动效落点**：

| 动效 | 落点 | 效果 |
|---|---|---|
| 光随指动 | 3D 视口玻璃覆盖层 + 悬浮信息卡 | 拖动旋转时高光跟随触点滑动，强化"触到实物"沉浸感 |
| 光线勾勒 | TitleBar + 主操作按钮（收藏/详情） | 动态光描边，深色背景下边界识别 |
| 非线性形变 | 底部悬浮商品卡 | 弹性弹出/收起，立体感 |

- **材质预设**（小件精品四类，商家扫描后选一个，存进模型元数据）：

| 预设 | 适用 | 参数取向 |
|---|---|---|
| 🔩 金属 | 饰品/金属潮玩 | 低粗糙度 + 高金属度，锐高光 |
| 💎 宝石 | 宝石类 | 高透射 + 折射 + 色散闪烁 |
| 🥃 玻璃 | 玻璃摆件 | 透射 + 轻微着色 |
| 🧶 哑光 | 手办/树脂 | 高粗糙度，柔漫射 |

### ⑤ Shell（模式切换/导航/全局光感）· ArkTS

- 单 App 双角色（买家⇄商家）；列表页 / 详情页 / 扫描页 / 我的
- 全局 TitleBar / TabBar 施加沉浸光感
- **不含交易**（P3）

## 5. 错误处理（仅校验系统边界，内部模块互信）

| 类别 | 处理 |
|---|---|
| 设备/能力不支持 | 启动时探测 API 26 + AR Engine 可用性；不支持→降级页"本设备不支持重建，仅可浏览" |
| 相机权限 | 标准授权流，拒绝时带说明二次引导 |
| AR Engine 初始化失败 | 报错 + 重试入口（部分设备无 AR Engine） |
| 重建失败 `SPATIAL_RECON_STATUS_FAILED` | 不崩溃，回扫描引导（"放慢环绕/补光"） |
| 温升过热 | 自动 `PauseSession` → "降温中" UI → 降温后 `ResumeSession`；持续过热→"保存当前进度退出" |
| 存储空间不足 | 扫描前预检可用空间；写入失败→保存已有 + 告警 |
| 模型加载/损坏/OOM | Showcase 错误态，回退缩略图 + "加载失败"；大模型内存压力→拒绝加载并提示 |
| napi 桥接异常 | NDK 错误封成类型化错误（`CaptureError` / `ReconError`）回 ArkTS，绝不裸崩 |

## 6. 测试策略

- **NDK 模块（CaptureKit/ReconEngine）**：mock 掉 AR Engine/Spatial Recon 调用，单测 napi 包装逻辑。真实重建只能在 **AGC 云调试真机**（API 26 / 7.0.0.23）做设备端集成测试——模拟器跑不了真 AR。
- **ModelStore**：真实本地 RelationalStore（临时库）单测；上传 stub 测契约。
- **Showcase 渲染**：NDK 渲染难单测→设备端截图快照 + 材质预设肉眼 QA。
- **光感 chrome / 手势**：ArkTS 组件级测试 + 模拟触摸。
- **Prove-it 闭环测试（最高优先级）**：录制一组 AR 帧作为 fixture → 喂 ReconEngine → 存 PLY → `ModelStore.list` → `Showcase.load` → 截图断言。**这条链路是 P1 的命脉，优先保它。**
- **温升/权限/不支持设备**：难自动化，覆盖于手工 QA checklist。

## 8. 风险与未决问题

1. **3DGS 模型体积/内存**：小件精品重建产物可能仍较大，低端机 OOM 风险。P1 用"拒绝加载并提示"兜底；流式/LOD 优化推迟。
2. ~~沉浸光感动效在 3D 覆盖层上的真实表现~~ **已澄清（2026-06-23）**：沉浸光感官方施加点是 `HdsNavigation` 标题栏与 `HdsTabs` 底部页签（`systemMaterialEffect`），非 XComponent 覆盖层。采取"外围 chrome"分支——光感落在外围 HDS chrome，3D 视口之上的悬浮卡用 `backgroundBlurStyle` 互补。详见 `docs/notes-immersive-light-sense.md`。残留待验证：`IMMERSIVE` 材质在真机的实际光影表现与四材质预设的契合度。
3. **材质预设与 3DGS 模型的契合**：3DGS 是重建产物而非传统 PBR 材质，"宝石/玻璃"预设能否真实呈现透射/折射待真机验证；不行则降级为环境光照风格切换。
4. **MP4 运镜视频价值**：P1 仅本地保存，是否值得生成待评估；若无人看，P2 砍掉。
5. **扫描引导有效性**：欠覆盖是重建质量头号杀手，引导 UI 的实际效果需用户测试校准。
